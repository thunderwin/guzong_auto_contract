# AI 合同助手执行计划（无数据库 / 无登录 / 全中文 / 第三方中转 AI）

## 1. 项目目标

构建一个中文 AI 助手：
1. 输入老板与对方的谈话记录（文本粘贴或文件导入）。
2. 自动提炼双方已达成条件、责任、付款、违约、争议解决等关键条款。
3. 输出可直接修改并用于签署盖章的合同草案（含风险提示与缺失项提醒）。

## 2. 关键约束与边界

1. 不使用数据库：所有会话与草稿仅保存在用户浏览器本地（`localStorage` / `IndexedDB`）。
2. 不需要登录：移除现有认证流程，首页直接可用。
3. 全中文界面：所有 UI 文案、提示、错误信息统一中文。
4. 模型来源：使用第三方中转 AI（OpenAI 兼容接口或自定义网关），不依赖官方直连。
5. 合规边界：
   - AI 只能生成“合同草案 + 风险提示”，不能承诺“自动具备法律效力”。
   - 在中国大陆法域下，`《中华人民共和国合同法》已于 2021-01-01 被《民法典》合同编取代`，建议以《民法典》合同编与《电子签名法》作为校验基线，并在上线前由执业律师复核模板。

## 3. 目标系统方案

### 3.1 核心流程

1. 用户输入对话记录。
2. AI 第 1 轮：结构化抽取（输出 JSON：主体、标的、金额、履约、违约、争议条款等）。
3. 规则引擎：检查必填条款是否缺失，输出“需补充项”。
4. AI 第 2 轮：基于结构化数据生成完整中文合同草案。
5. 前端合同编辑器：用户可逐条改写。
6. 导出文件：PDF / DOCX（用于打印、签署、盖章）。

### 3.2 技术架构

1. 前端：Next.js（保留现有框架），新增“合同工作台”页面。
2. 后端 API：仅保留无状态 AI 调用接口，不保存业务数据。
3. 存储：浏览器本地存储草稿、最近记录、模板偏好。
4. AI 接入：第三方中转网关（`BASE_URL + API_KEY + MODEL`）。
5. 输出控制：强制 JSON Schema + 合同模板填充，降低模型自由发挥。

## 4. 基于当前仓库的改造清单

1. 认证移除：
   - 删除或停用 `app/(auth)/*`。
   - 去除 `app/layout.tsx` 中 `SessionProvider` 依赖。
   - 删除前端登录/注册入口与侧边栏用户态依赖。
2. 数据库移除：
   - 停用 `lib/db/*` 与所有 `@/lib/db/queries` 引用。
   - 改写 `app/(chat)/api/*` 中依赖会话/数据库的接口。
   - `package.json` 去掉构建前迁移（`tsx lib/db/migrate`）。
3. AI 路由重构：
   - 新增 `/api/contract/extract`（对话 -> 结构化要点）。
   - 新增 `/api/contract/generate`（要点 -> 合同文本）。
   - 新增 `/api/contract/check`（合规检查与缺失项提示）。
4. 前端重构：
   - 聊天 UI 改为“对话输入 + 条款面板 + 合同编辑器 + 导出”。
   - 全量中文文案替换。

## 5. 分阶段执行（建议 7~10 个工作日）

### 阶段 A：需求与法务基线（0.5~1 天）

1. 确认合同类型优先级（采购/服务/销售/合作）。
2. 明确目标法域与强制条款清单。
3. 固化“AI 非法律意见”提示语与人工复核节点。

交付物：`合同条款清单 + 风险提示清单 + 模板目录`。

### 阶段 B：系统去耦（1~2 天）

1. 下线登录与访客体系。
2. 移除数据库读写与历史记录 API。
3. 将本地缓存改为浏览器存储。

交付物：可运行的“无登录、无数据库”基础版本。

### 阶段 C：AI 合同引擎（2~3 天）

1. 编写抽取提示词（结构化 JSON 输出）。
2. 编写合同生成提示词（模板化、可编辑）。
3. 编写条款检查提示词（缺失项 + 风险等级）。
4. 接入第三方中转 AI 参数配置与重试机制。

交付物：三段式合同生成链路（抽取/生成/检查）。

### 阶段 D：中文合同工作台（1~2 天）

1. 构建中文操作流页面。
2. 增加“关键条款确认”步骤，避免误生成。
3. 支持导出 PDF/DOCX。

交付物：可直接给业务演示的 UI。

### 阶段 E：测试与上线（1~2 天）

1. 准备 20~30 组真实/脱敏对话样例回归。
2. 做合同完整性、稳定性、错误处理、导出可用性测试。
3. 部署并配置运行监控（仅接口性能与错误率，不记录业务正文）。

交付物：上线版本 + 验收报告。

## 6. 验收标准（DoD）

1. 用户无需登录即可完整使用全流程。
2. 项目运行不依赖 `POSTGRES_URL` 等数据库变量。
3. 所有核心页面与提示为中文。
4. 同一输入可稳定产出“结构化条款 + 合同草案 + 缺失项清单”。
5. 导出的合同文档可被直接修改并用于签署流程。
6. 明确展示“需法务复核”提示，不宣称 AI 自动提供法律结论。

## 7. 运行配置（建议）

建议新增环境变量（命名可按你的中转服务调整）：

1. `AI_PROXY_BASE_URL`：第三方中转地址。
2. `AI_PROXY_API_KEY`：第三方中转密钥。
3. `AI_PROXY_MODEL`：默认模型（如 `deepseek-chat` / `qwen-max` 等）。
4. `AI_TIMEOUT_MS`：请求超时毫秒数（建议 30000）。

## 8. 风险与应对

1. 模型幻觉导致条款错误：使用“结构化抽取 + 条款确认 + 缺失项检查”三重约束。
2. 合同法律风险：模板先经律师审定，页面强制显示法务复核提示。
3. 第三方中转不稳定：增加重试、超时、降级提示（保留用户输入不丢失）。
4. 本地存储风险：提供“导出备份/导入恢复”功能，避免浏览器清理导致数据丢失。

---

该计划按“先去耦（无登录/无数据库）再做合同能力”推进，可在短周期内做出可签署流程的业务版本；若要提升法律稳健性，下一步重点是“律师审定模板 + 行业条款库”。
